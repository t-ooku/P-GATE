# 0. エグゼクティブ・ディレクティブ

CORE  MYGATEはAI検索アプリではない。『欲しい』を入口に、対話・記憶・分析・通知・購入・学習まで伴走するプラットフォームである。

Project GATEは既存実装の開発コードネーム、MYGATEは対外サービス名称である。GAS、Power Automate Desktop、Google Drive、Spreadsheet、GitHubなどの既存資産は監査して再利用するが、UI・ブランド・ユーザー向け名称はMYGATEへ統一する。

Codexは実装責任者として働く。独自仕様を増やすのではなく、本書と検証済み実装を整合させ、品質・安全性・保守性・拡張性を高める。不明点は推測で固定せず、明示的な検証項目にする。

| 区分 | 定義 | 扱い |

| --- | --- | --- |

| 確定仕様 | ユーザーが明示的に決定したブランド・機能・方針 | 実装基準。変更は承認制 |

| 確認済み実績 | ログ・画面・実行結果で確認された事実 | 再現テストで維持 |

| 報告済み／未監査 | 参照会話で語られたが現リポジトリで未確認 | 監査後に昇格または棄却 |

| 将来構想 | 価値仮説・ロードマップ上の候補 | MVPへ混入させない |

# 2. サービス思想と差別化

DESIGN PHILOSOPHY  MYGATEは『思い出す前の検索』を実現する。商品名を知らなくても、形・用途・記憶・状況から一緒に探す。

| 一般的なAIアプリ | MYGATE |

| --- | --- |

| 答えを返す | 欲しいを叶える |

| 会話で終わる | 会話から始まる |

| 今だけ答える | 見つかるまで覚える |

| 検索する | 一緒に探す |

| ユーザーが再検索する | AIが知らせる |

| 商品を教える | 購入まで伴走する |

## PH-001  ジャンル非限定

| 項目 | 仕様 |

| --- | --- |

| 優先度 | MUST |

| 要求 | 初期公開から特定ジャンルにブランドを固定しない。AIが会話からカテゴリを推定し、自然に需要クラスタを形成する。 |

| 受入条件 | ユーザーはカテゴリを選ばずに検索を開始でき、結果・ログでは分類根拠と信頼度を保持する。 |

## PH-002  生活知識への拡張

| 項目 | 仕様 |

| --- | --- |

| 優先度 | MUST |

| 要求 | 互換性、海外販売、代替品、価格、使い方等の生活上の疑問も扱う。医療等の高リスク領域は回答制限と一次情報誘導を必須にする。 |

| 受入条件 | 高リスク領域で断定・診断・服薬指示を行わず、信頼できる一次情報と専門家確認を促す。 |

## PH-003  購買中立性

| 項目 | 仕様 |

| --- | --- |

| 優先度 | MUST |

| 要求 | 最安値・スポンサー・送客収益が回答品質を歪めない。オーガニック結果と広告を明確に分離する。 |

| 受入条件 | スポンサー表示、ランキング根拠、価格の取得時刻、販売元が表示される。 |

# 3. 利用者と主要ジョブ

| 主体 | 課題 | MYGATEの仕事 |

| --- | --- | --- |

| 一般ユーザー | 商品名が分からない／記憶が曖昧／比較が面倒 | 自然な会話で候補を絞り、見つからなければMYWISHへ |

| 無料会員 | 後で確認したい／値下げを待ちたい | お気に入り、履歴、MYWATCH、パーソナライズ |

| セラー | 売れる商品を早く見つけたい | Opportunity、MYTREASURE、コンテンツ、SNS、送客 |

| メーカー | 未充足需要を商品企画へ変えたい | Treasure Map、待機者、実現性、発売通知 |

| 代理店 | 複数顧客を効率運用したい | 権限分離、ポートフォリオKPI、導入支援 |

| 運営 | 成長とコスト・安全性を両立したい | 利用量制御、監査、透明な収益化、SLO |

## 3.1 非会員と無料会員

- 非会員にもAI検索・Knowledge閲覧・ランキング・比較・商品・動画を広く提供する。濫用防止のレート制限は透明に表示する。

- 無料会員は個人情報の最小化を原則とし、MYWISH、MYWATCH、履歴、パーソナライズ、通知を利用できる。

- 無料会員化の価値は追跡・再通知であり、検索結果を見るための強制登録にしない。
